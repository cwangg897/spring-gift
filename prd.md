## 기능 요구 사항
현재 있는 코드에서 리펙토링을 목표로한다
- 누구나 동일하게 실행 가능하고 프로덕션과 유사한 환경에서 테스트할 수 있는 시스템을 구축한다. 개인 로컬 설정에 의존하지 않고 반복 실행할 수 있는 자동화된 테스트 환경을 만든다.
- 구조 변경을 통해 변경 난이도를 낮추되, 기존 작동은 유지한다. 
  - 스타일 정리: 불필요한 코드 제거 (작동 변경 없음)
  - 서비스 계층 추출: 구조 변경 (작동 변경 없음)

- 작동 변경을 안전하게 수행하고, 그 결과를 증거로 보여준다. 
  - 트랜잭션 경계 세우기 
  - 누락된 작동 구현 
  - 도메인 책임 되찾기

- 각 리펙토링을 도메인 단위로 진행한다
  - 도메인 마다 진행했을 때 변경사항을 루트폴더에 진행상황폴더를 만들고 거기에 진행상황에 대해 문서화를 한다 
  - 문서화는 모두 통일되도록 진행한다

- 범위는 /spring-gift/src/main/java에 있는 파일들이다 즉 자바 파일들이다

## 아키텍처 
- 레이어드 아키텍쳐로 선정한다(controller, service, repository, entity, dto, repository)

## 테스트 환경
- 테스트 환경은 테스트 컨테이너를 사용한다


## 프로그래밍 요구 사항
- TDD 루프를 유지한다.
  - 가능하면 Red, Green, Refactor 순서로 진행한다.
  - 최소 기준은 "변경 후 전체 테스트 통과"다.

- 한 번에 한 조각만 변경한다.
  - "다음 변경 1개"처럼 범위를 제한한다.
  - AI에게도 현재 변경 범위를 명확히 제한한다.

- ADR은 필요시 작성한다.
  - 다음 조건 중 하나라도 해당하면 ADR을 남긴다.
    - 선택지가 2개 이상이고 트레이드오프가 존재한 경우
    - 반복적으로 따라야 할 규칙이나 경계를 정의한 경우
    - 테스트 전략이나 검증 방식이 결정의 핵심이었던 경우
